## Scope

Review all `.rs`, `.c`, `.h`, `Cargo.toml`, `bird.conf` files. Ignore
autogenerated files (`erezd/src/bpf/erez.skel.rs`,
`erezd/src/bpf/vmlinux.h`) and everything under `target/`, `.git/`,
`node_modules/`. Also ignore non-workspace files, i.e. `netgauze/`.

When reviewing a change, apply all rules to the changed files. For consistency
rules (R7, R7.2, R11), also scan the wider codebase. When performing a full
audit, review all files in scope.

## How to review

Apply every rule below. Use diff format for concrete fixes:

    **R1** `erezd/src/bpf.rs:42`
    ```diff
    - // send updates about NLRIs via this channel
    + /// Channel for sending NLRI updates.
    ```

For cross-file findings, use a plain description:

    **R7.1** `bgp.rs:34` vs `bpf.rs:91`
    Finding: `iface` in bgp.rs but `interface` in bpf.rs. Prefer `iface`.

Produce a summary count grouped by file at the end.

Do not output any findings which do not contain violations of the rules!

## Rules

### R1 Comments are proper sentences

Comments must be capitalised (apart from lowercase literals) and end with a
full stop, or some other form of punctuation, i.e. exclamation mark. Also,
bulleted points don't need to end with full stops.

Bad: `// send updates about NLRIs via this channel`
Good: `/// Channel for sending NLRI updates.`

### R2 Plain strings over rustdoc links

Bad: `` /// Interface name, (e.g. [`bra401`]). ``
Good: `/// Interface name, (e.g. "bra401").`

### R3 Rust/C struct sync comments

Any Rust struct with a C counterpart (or vice versa) must have a comment
noting the pairing, for example:

- `// Keep in sync with its C counterpart`
- `// It must be kept in sync with the C structure`
- `// It must be kept in sync with the C enum`

Exception: Rust wrappers around C standard-library / POSIX types (e.g.
`in6_addr`, `sockaddr_in6`) are excluded. Their layout is fixed by the OS
ABI, so there is no project-owned C definition to sync with.

### R4 No blank lines between struct fields or enum variants

```diff
  NotAttached { iface: String },
-
  #[snafu(transparent)]
  CData {
```

### R5 Error messages must be capitalised

Applies to `#[snafu(display(...))]`, `.whatever_context()`, `.context()`,
`anyhow!`, `bail!`. Exception: lowercase literals.

Bad: `.whatever_context("failed to load config")`
Good: `.whatever_context("Failed to load config")`

### R6 Log messages must be capitalised

Applies to all `tracing` macros. Exception: messages starting with a literal
lowercase identifier (crate name, function name, variable name).

Bad: `info!("starting bgp speaker")`
Good: `info!("Starting BGP speaker")`

### R7 Naming consistency

Check for naming consistency across the entire codebase. Perform a
codebase-wide search for variable, parameter, field, and function names.
The list below is not exhaustive -- flag any inconsistency.

#### R7.1 General naming

- `addr` vs `address` -- `addr` in code, `address` in user-facing text.
- `Err(e)` vs `Err(err)` -- prefer `Err(e)`.
- `res` vs `result` -- prefer `result`, or a descriptive name.
- `Self` vs concrete type -- prefer the concrete type name.
- `ok_or` vs `ok_or_else` -- `ok_or` for cheap values, `ok_or_else` for
  computed ones.

#### R7.2 Parameter names across call sites

The same logical value must have the same parameter name everywhere.

```diff
- fn detach(interface: &Interface) -> ...
+ fn detach(iface: &Interface) -> ...      // matches attach(iface: ...)
```

### R8 Consistent parameter ordering

Shared parameters must appear in a consistent order:

1. Primary argument.
2. Configuration/options.
3. Logger/tracing span.
4. Cancellation token (always last).

Exception: when the method's purpose is to operate on the token or logger.

### R9 Test assertions

- Assertion messages: non-capitalised, no full stop (unless referring to an
  uppercased literal).
- `assert_eq!`/`assert_neq!`: `(actual, expected)`, not `(expected, actual)`.

### R10 Clone at the closure boundary

Clone values where they are moved into a closure, not earlier in the function.

```diff
- let token2 = token.clone();
- // ... 20 lines ...
- tokio::spawn(async move { token2.cancelled().await });
+ tokio::spawn({
+     let token = token.clone();
+     async move { token.cancelled().await }
+ });
```

### R11 Consistent qualification

Use the same level of qualification for the same item everywhere. Do not mix
`bpf::attach()`, `crate::bpf::attach()`, and bare `attach()`. Do not mix
`anyhow!` and `anyhow::anyhow!`.

### R12 British English spelling

Use British English in all comments, docs, error messages, and logs.

| American | British |
|----------|---------|
| optimize | optimise |
| behavior | behaviour |
| color | colour |
| initialize | initialise |
| center | centre |
| canceled | cancelled |
| license | licence (noun) |
| analyze | analyse |
| organize | organise |

### R13 C/BPF signed return types

If a C function can return `-1`, its return type must be signed.

Bad: `static __always_inline u64 some_func(...)` (returns -1 on error).
Good: `static __always_inline s64 some_func(...)`.

### R14 Dependencies

- Move repeated dependencies to `[workspace.dependencies]`.
- Remove unused dependencies.
- Test-only code should use `[dev-dependencies]`.
- Crate-specific features should not be in workspace-level config.

### R15 Structured logging fields

Use consistent field names in tracing events. Errors use Display
(`error = %e`), complex types use Debug (`?config`).

Bad: `warn!(err = %e, "Failed")` -- use `error`, not `err`.
Good: `warn!(error = %e, "Failed parsing BGP update message")`
